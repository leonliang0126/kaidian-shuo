# 《开店说》员工系统重构 — 简单 PRD

> 作者：许清楚（PM）｜ 版本：v1.0 ｜ 日期：2026-07-08
> 项目：KaiDianShuo v3 ｜ 技术栈：Vite + React 18 + TypeScript strict + Tailwind CSS + Zustand

---

## 1. 产品目标

**一句话目标**：用有属性的真实员工团队替换掉粗暴的 staffTier 滑块，让玩家体验"招人—排班—士气—裁人"的完整用工链条，用人决策成为经营策略的核心维度。

---

## 2. 用户故事

| # | 角色 | 愿望 | 收益 |
|---|------|------|------|
| US1 | 玩家 | 我想在生意差时少排几天班来降人力成本，而不是只能切一个固定的档位 | 弹性控制日成本，细粒度经营 |
| US2 | 玩家 | 我想看到每个员工的属性（7天后）来安排最合适的排班 | 针对性地用好人，而不是闭眼排 |
| US3 | 玩家 | 我想在生意火爆时花行动点多招一个人，而不是被 fixed capacity 卡死 | 用人决策和经营节奏联动 |
| US4 | 玩家 | 我想看到士气变化，在员工不开心时安排放假或发奖金 | 防止突发离职/罢工，感受管理深度 |
| US5 | 玩家 | 我想裁掉摸鱼王/刺头/关系户，但要面对补偿金和品牌风险 | 让裁员有真实代价，不做无脑操作 |

## 5. 数据结构建议

### 5.1 `Employee` 类型（新增）

```typescript
// src/types/employee.ts

export type EmployeeAttribute =
  | 'super_worker'      // 超级卷王
  | 'old_smooth'        // 老油条
  | 'slacker'           // 摸鱼王
  | 'rookie'            // 新手上路
  | 'all_rounder'       // 全能选手
  | 'troublemaker'      // 刺头
  | 'social_butterfly'  // 社交蝴蝶
  | 'guanxi_hire';      // 关系户

export interface Employee {
  id: string;
  name: string;
  joinDay: number;                  // 入职 day
  attribute: EmployeeAttribute;     // 真实属性
  isExposed: boolean;               // 是否已暴露（入职 >= 7天 → true）
  morale: number;                   // 士气 0-100
  monthlySalary: number;            // 月薪
  daysWorkedThisWeek: number;       // 本周已上班天数
  isScheduledToday: boolean;        // 今日是否排班
  weeklyWorkDays: number[];         // 本周每天排班记录 [day1, day2, ...]
  consecutiveWorkDays: number;      // 连续工作天数（用于长期不放假检测）
  isTempStaff: boolean;             // 是否为事件临时员工
}
```

### 5.2 `EmployeeAttributeConfig`（属性配置表）

```typescript
// src/data/employeeAttributes.ts
export interface EmployeeAttributeConfig {
  type: EmployeeAttribute;
  label: string;
  efficiencyRange: [number, number];  // 效率系数范围 [min, max]
  specialMechanic: string;            // 特殊机制描述
  descriptionHints: string[];          // 隐晦自介绍的随机文案池
}
```

### 5.3 关联 StoreState / GameState 变更

**StoreState 移除字段**：
- `staffTier` → 删除
- `capacity` → 改为由当日排班员工数动态计算
- `efficiency` → 改为员工效率系数的聚合值

**StoreState 新增字段**：
```typescript
// 每个店面有独立的员工列表
employees: Employee[];
// 今日承载上限 = 今日排班员工数 × 基准承载系数
// 基准承载系数：可配置常量（如 70 单/人/天）
```

**GameState 新增字段**：
- 暂无全局级字段（员工挂在 StoreState 上，每个店独立）

### 5.4 员工工资计算公式

```
每日总staffCost = Σ(今日排班员工.monthlySalary / 30)
                  × (该员工今日超时加班 ? 1.5 : 1.0)
                  + mods.staffCostAdd
                  × (1 + mods.staffCostPct / 100)
```

### 5.5 排班承载公式

```
effectiveCapacity = 今日排班人数 × BASE_CAPACITY_PER_STAFF
其中 BASE_CAPACITY_PER_STAFF = 70（可调）
```

## 7. 待确认问题

| # | 问题 | 建议 / 思考 |
|---|------|-------------|
| Q1 | **装修档 ↔ 最大员工数映射表** | 建议：bare=2人, clean=3人, memorable=4人, viral=5人, designer=6人。待 PM 与数值策划确认具体数字 |
| Q2 | **刷新候选人是否消耗行动点** | 需求提到"可选"。建议：**首次打开免费**，后续每次刷新消耗 1 行动点。这样抑制反复刷属性的行为 |
| Q3 | **裁人补偿金 = N 天工资的 N 值** | 建议 N=10（10 天工资）。可调，需 playtest 校准 |
| Q4 | **基准承载系数 BASE_CAPACITY_PER_STAFF** | 建议 70 单/人/天。实际还需要参考旧 staffTier capacity 值来反推（五档: 90/140/280/330/450 → 对应 1~5人 → 每人约 70~90 单） |
| Q5 | **员工月薪基准值** | 建议不同店型不同基准（奶茶店低、咖啡店高）。先用固定基准 ¥5,000 ±20% 浮动，后续分店型校准 |
| Q6 | **假期系统** | 玩家是否可主动安排"全员放假一天"（不排班但员工士气恢复）？建议实现 |
| Q7 | **关系户出现概率** | 建议低概率（~5%），作为稀有的"双刃剑" |
| Q8 | **多店员工管理** | 本期先做单店（主店）；多店的员工系统放在后续迭代 |
